# PRD: 맛집 저장 및 추천 모바일 앱

## 1. 개요

### 1.1 제품 요약
사용자가 방문한 식당을 저장하고 관리하며, 위치 기반으로 주변 맛집을 추천받을 수 있는 모바일 앱.

### 1.2 배경 및 문제 정의
- 사용자는 방문한 맛집을 메모장, 카카오톡 등 여러 곳에 흩어져 관리하고 있음
- 점심시간 등 식사 장소를 결정할 때 매번 고민이 반복됨
- 기존 맛집 앱(망고플레이트, 식신 등)은 타인의 리뷰 중심이며 **개인 맛집 아카이빙**에 특화되지 않음

### 1.3 제품 비전
> "나만의 맛집 지도를 만들고, 오늘 뭐 먹을지 고민하지 않는다."

### 1.4 목표
| 목표 | 지표 | 타겟 (출시 후 3개월) |
|------|------|---------------------|
| 사용자 확보 | MAU | 10,000명 |
| 핵심 기능 활용 | 사용자당 평균 저장 맛집 수 | 15개 이상 |
| 리텐션 | D7 Retention | 40% 이상 |
| 추천 기능 활용 | 일일 추천 요청 수 / DAU | 0.5회 이상 |

## 2. 타겟 사용자

### 2.1 주요 페르소나

#### 페르소나 A: 직장인 김점심 (28세)
- **상황**: 강남 오피스 근무, 매일 점심 메뉴 고민
- **행동**: 맛집을 발견하면 카카오톡 나에게 보내기로 저장, 나중에 찾기 어려움
- **니즈**: 회사 근처 저장한 맛집 중 빠르게 하나를 추천받고 싶음
- **핵심 시나리오**: 점심시간 5분 전, 앱에서 "오늘 뭐 먹지?" 버튼 한 번으로 추천받기

#### 페르소나 B: 맛집 탐험가 이푸디 (33세)
- **상황**: 주말마다 새로운 맛집 탐방, SNS에 기록
- **행동**: 방문한 식당에 대한 개인 메모와 사진을 체계적으로 관리하고 싶음
- **니즈**: 카테고리별 정리, 사진 첨부, 개인 평점 관리
- **핵심 시나리오**: 방문 후 식당 정보와 사진, 한줄평을 빠르게 기록

### 2.2 타겟 시장
- **1차 타겟**: 20~30대 직장인 (점심 추천 니즈)
- **2차 타겟**: 맛집 탐방을 즐기는 20~40대 (아카이빙 니즈)
- **지역**: 한국 (서울/수도권 우선)

## 4. 상세 기능 요구사항 (P0)

### 4.1 F-001: 회원가입/로그인

**사용자 스토리**
> 사용자로서, 이메일과 비밀번호로 회원가입 및 로그인하여 내 데이터를 안전하게 관리하고 싶다.

**요구사항**
| ID | 요구사항 | 상세 |
|----|---------|------|
| F-001-01 | 이메일 회원가입 | 이메일, 비밀번호, 닉네임 입력 |
| F-001-02 | 이메일 유효성 검증 | 올바른 이메일 형식 확인 |
| F-001-03 | 비밀번호 규칙 | 최소 8자, 영문 + 숫자 + 특수문자 포함 |
| F-001-04 | 이메일 중복 확인 | 기존 가입 이메일 실시간 체크 |
| F-001-05 | 로그인 | 이메일 + 비밀번호 로그인 |
| F-001-06 | 자동 로그인 | 앱 재실행 시 토큰 기반 자동 로그인 |
| F-001-07 | 로그아웃 | 설정 화면에서 로그아웃 |

**예외 처리**
- 잘못된 이메일/비밀번호: "이메일 또는 비밀번호가 올바르지 않습니다."
- 이미 가입된 이메일: "이미 사용 중인 이메일입니다."
- 네트워크 오류: "네트워크 연결을 확인해주세요."

### 4.2 F-002: 식당 추가

**사용자 스토리**
> 사용자로서, 방문한 식당의 정보를 입력하여 내 맛집 리스트에 저장하고 싶다.

**요구사항**
| ID | 요구사항 | 상세 |
|----|---------|------|
| F-002-01 | 식당 이름 입력 | 필수, 최대 200자 |
| F-002-02 | 주소 입력 | 필수, 주소 검색 또는 직접 입력 |
| F-002-03 | 위치 좌표 자동 설정 | 주소 입력 시 위도/경도 자동 변환 |
| F-002-04 | 카테고리 선택 | 1개 이상 선택, 복수 선택 가능 |
| F-002-05 | 개인 평점 | 선택, 1~5점 (별점) |
| F-002-06 | 메모 | 선택, 최대 1,000자 |
| F-002-07 | 전화번호 | 선택, 자동 전화 연결 지원 |
| F-002-08 | 사진 첨부 | 선택, 최대 5장, 장당 10MB 이하 |

**UI 흐름**
```
[+ 식당 추가] 버튼 탭
    → 식당 이름 입력
    → 주소 검색 (검색 결과에서 선택 또는 직접 입력)
    → 카테고리 선택 (칩 형태, 복수 선택)
    → 평점 (별점 슬라이더)
    → 메모 (텍스트 영역)
    → 사진 추가 (카메라/앨범)
    → [저장] 버튼
```

### 4.4 F-007: 위치 기반 추천

**사용자 스토리**
> 사용자로서, 현재 위치 주변에서 오늘 점심으로 갈 만한 식당을 추천받고 싶다.

**요구사항**
| ID | 요구사항 | 상세 |
|----|---------|------|
| F-007-01 | 현재 위치 탐지 | GPS 기반 현재 위치 확인 |
| F-007-02 | 반경 설정 | 기본 1km, 사용자 조절 가능 (500m ~ 5km) |
| F-007-03 | 추천 결과 | 1~3개 식당 추천 (기본 3개) |
| F-007-04 | 추천 재요청 | "다른 추천 보기"로 새로운 결과 요청 |
| F-007-05 | 카테고리 제외 | 특정 카테고리 제외 옵션 ("오늘은 일식 빼고") |
| F-007-06 | 추천 알고리즘 | 아래 참조 |

**추천 알고리즘 (v1)**
```
입력: 사용자 위치 (lat, lng), 반경, 제외 카테고리
처리:
  1. 반경 내 저장된 식당 필터링
  2. 최근 3일 내 방문한 식당 제외
  3. 스코어 계산:
     - 평점 가중치: rating / 5 × 0.4
     - 거리 가중치: (1 - distance / radius) × 0.3
     - 카테고리 다양성: 최근 5회 방문과 다른 카테고리 → +0.1
     - 랜덤 요소: random(0, 0.2)
  4. 스코어 상위 N개 반환
출력: 추천 식당 리스트 (이름, 거리, 카테고리, 평점)
```

**UI 흐름**
```
[오늘 뭐 먹지?] 버튼 탭
    → 위치 권한 확인 (미허용 시 권한 요청)
    → 로딩 애니메이션 (1~2초)
    → 추천 카드 3장 표시 (스와이프 가능)
    → [이 식당으로 결정!] → 방문 기록 업데이트
    → [다른 추천 보기] → 새로운 3개 추천
```

## 5. 비기능 요구사항

|------|------|

| 통신 | HTTPS 필수 (TLS 1.2+) |
| 이미지 업로드 | Presigned URL 사용, 파일 타입/크기 서버 검증 |
| 개인정보 | 위치 정보 수집 동의, 개인정보 처리방침 필수 |

### 5.3 호환성
| 항목 | 요구사항 |
|------|---------|
| iOS | 15.0 이상 |
| Android | API 26 (8.0) 이상 |
| 화면 크기 | iPhone SE ~ iPhone 16 Pro Max, 일반적인 안드로이드 디바이스 |

### 5.4 확장성
- 동시 접속 1,000명 기준 설계
- MVP 이후 수평 확장 가능한 아키텍처

## 7. 데이터 모델 (핵심 엔티티)

```
User
├── id, email, password, nickname, profileImageUrl
├── latitude, longitude (기본 위치)
└── createdAt, updatedAt

Restaurant
├── id, userId, name, address
├── latitude, longitude
├── phone, memo, rating (1-5)
├── lastVisitedAt
└── createdAt, updatedAt

Category
├── id, userId, name, colorHex, sortOrder
└── createdAt, updatedAt

Review
├── id, userId, restaurantId
├── content, rating, visitedDate
└── createdAt, updatedAt

Image
├── id, restaurantId, userId
├── url, thumbnailUrl, s3Key
├── fileSize, mimeType, sortOrder
└── createdAt
```

**관계**
- User 1:N Restaurant (사용자가 식당 저장)
- Restaurant N:M Category (식당은 여러 카테고리, 카테고리는 여러 식당)
- Restaurant 1:N Review (식당에 여러 리뷰)
- Restaurant 1:N Image (식당에 최대 5장 이미지)

## 10. 리스크 및 대응

| 리스크 | 영향 | 대응 방안 |
|--------|------|----------|
| 위치 권한 미허용 | 추천 기능 사용 불가 | 수동 위치 입력 폴백, 권한 필요성 안내 UI |
| 오프라인 동기화 충돌 | 데이터 불일치 | 타임스탬프 기반 Last-Write-Wins + 충돌 알림 |
| 이미지 용량 증가 | 스토리지 비용 증가 | 클라이언트 압축, 썸네일 자동 생성, S3 Lifecycle 정책 |
| 초기 콘텐츠 부족 | 추천할 식당이 없음 | 온보딩에서 인기 맛집 5개 기본 추가 유도 |
| 주소/좌표 변환 실패 | 식당 위치 부정확 | 지도 핀 수동 조정 기능 제공 |

## 11. 범위 외 (Out of Scope)

MVP에서 **포함하지 않는** 기능:
- 사용자 간 소셜 기능 (친구, 팔로우, 공유 맛집 리스트)
- 식당 예약 연동
- 결제/쿠폰 기능
- AI 기반 고도화 추천 (협업 필터링 등)
- 웹 버전
- 다국어 지원 (한국어 전용)
- 식당 사장님 대상 기능 (B2B)

## 부록 A: 용어 정의

| 용어 | 정의 |
|------|------|
| 식당 | 사용자가 저장한 음식점 (레스토랑, 카페, 술집 등 모두 포함) |

| 오프라인 모드 | 네트워크 없이 로컬 데이터로 앱을 사용하는 상태 |
